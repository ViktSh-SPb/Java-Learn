### Что такое REST?

REST (Representational State Transfer) — это архитектурный стиль для разработки распределённых систем, в частности веб-сервисов. Он основывается на использовании стандартных протоколов HTTP и принципах простоты, масштабируемости и статeless (без сохранения состояния между запросами). В REST ресурсы идентифицируются URI, а взаимодействие осуществляется через стандартные HTTP-методы с использованием представлений ресурсов (например, JSON или XML).

### Перечислите какие-либо коды ответов HTTP и их значения

|Код ответа|Значение|Описание|
|---|---|---|
|200 OK|Успешный запрос|Запрос выполнен успешно, и в ответе содержится запрошенная информация.|
|201 Created|Создано|Новый ресурс успешно создан (обычно после POST).|
|204 No Content|Нет содержимого|Запрос выполнен успешно, но возвращать ничего не нужно (часто после DELETE или PUT).|
|400 Bad Request|Неверный запрос|Ошибка клиента: неправильный синтаксис или некорректные параметры.|
|401 Unauthorized|Не авторизован|Требуется аутентификация.|
|403 Forbidden|Запрещено|Нет прав на выполнение операции.|
|404 Not Found|Не найдено|Ресурс не существует.|
|500 Internal Server Error|Внутренняя ошибка сервера|Ошибка на стороне сервера.|

### Какие HTTP методы считаются идемпотентными?

Идемпотентные методы — такие, при повторном выполнении которых результат не изменится:

- GET — получение ресурса.
- PUT — создание или обновление ресурса по конкретному URI.
- DELETE — удаление ресурса.
- HEAD — получение метаданных ресурса без тела.
- OPTIONS — получение поддерживаемых методов и возможностей.

Метод POST обычно не считается идемпотентным, так как он часто создаёт новые ресурсы при каждом вызове.

### Как в контексте REST применяются принципы безопасности и идемпотентности?

- Безопасность: REST использует стандартные механизмы HTTP для обеспечения безопасности:
    - HTTPS для шифрования данных.
    - Аутентификация через OAuth, API ключи, JWT.
    - Контроль доступа и авторизация.
- Идемпотентность: Использование идемпотентных методов (GET, PUT, DELETE) помогает избежать нежелательных эффектов при повторных запросах, например, при сетевых сбоях или повторных отправках. Это повышает надёжность взаимодействия и упрощает обработку ошибок.

### Каково значение и использование кода ответа HTTP 204?

Код 204 No Content означает успешное выполнение запроса без возвращения содержимого. Обычно используется:

- После успешного удаления ресурса (DELETE).
- После успешного обновления данных (PUT), когда возвращать обновлённый ресурс не требуется. Это помогает снизить объем передаваемых данных и повысить эффективность.

### В чем разница между методами PUT и PATCH в контексте REST?

- PUT: полностью заменяет ресурс по указанному URI. Если ресурс отсутствует — создаёт его.
- PATCH: применяет частичные изменения к существующему ресурсу. Используется для обновления отдельных полей без необходимости пересоздавать весь ресурс.

Пример:

- PUT /users/123 с телом `{ "name": "John", "age": 30 }` заменит весь ресурс пользователя.
- PATCH /users/123 с телом `{ "age": 31 }` обновит только поле `age`.

### Как REST использует принципы единообразия интерфейса и беспостредственного управления ресурсами?

- Единообразие интерфейса: стандартизированные методы HTTP (GET, POST, PUT, DELETE) позволяют клиентам взаимодействовать с любыми ресурсами одинаковым образом.
- Беспостредственное управление ресурсами: каждый ресурс идентифицируется уникальным URI; клиент управляет состоянием ресурсов через стандартные операции без необходимости знать внутреннюю реализацию сервера.

Это упрощает разработку, тестирование и масштабирование систем.

### Какие существуют подходы к обработке версионирования в RESTful API?

- Включение номера версии в URL: `/api/v1/users`.
- Использование заголовков: `Accept: application/vnd.example.v1+json`.
- Передача версии как параметра запроса: `/api/users?version=1`.

Выбор подхода зависит от требований к совместимости и удобства использования.

### Опишите, как в RESTful веб-сервисах реализуется кеширование и его влияние на производительность

REST использует стандартные HTTP-заголовки для управления кешированием:

- `Cache-Control` — указывает политики кеширования (`public`, `private`, `no-cache`, `max-age`).
- `ETag` — уникальный идентификатор версии ресурса; клиент может использовать его для проверки актуальности данных.
- `Last-Modified` — дата последнего изменения; позволяет клиенту запрашивать только при изменениях.

Кеширование снижает нагрузку на сервер, уменьшает задержки и повышает масштабируемость системы.

### Объясните, что такое HATEOAS?

HATEOAS (Hypermedia As The Engine Of Application State) — это принцип REST, предполагающий включение гипермедиа ссылок внутри ответов API. Клиенты используют эти ссылки для навигации по API без необходимости знать структуру заранее. Это делает API более самодокументированным и гибким при изменениях.

Пример: при получении информации о заказе в ответе могут быть ссылки на связанные ресурсы: оплату, доставку или возврат.

# 1. **Сложность проектирования и разбиения**
- Нужно определить **границы микросервисов** (bounded contexts).
- Неправильное разбиение → сильная связанность, частые межсервисные вызовы.
- Сложно оценить, какие модули стоит вынести первыми.
# 2. **Повышенная сложность инфраструктуры**
- В монолите деплой простой, один сервер.
- В микросервисах появляются:
    - сеть между сервисами
    - контейнеризация (Docker)
    - оркестрация (Kubernetes)
    - Service Mesh (Istio, Linkerd)
- Требуется мониторинг, логирование и трассировка распределённых вызовов.
# 3. **Коммуникация между сервисами**
- Появляется риск **latency**, сбоев, дублирующих сообщений.
- Нужно выбирать синхронные (REST, gRPC) или асинхронные (Kafka, RabbitMQ) коммуникации.
- Необходимо управлять **eventual consistency** вместо транзакционной согласованности.
# 4. **Управление данными и транзакциями**
- Распределённые транзакции сложнее, чем в монолите.
- Нужно применять паттерны:
    - Saga (хореография/оркестрация)
    - Event Sourcing
    - Eventual Consistency
# 5. **Мониторинг и отладка**
- Ошибки становятся **распределёнными**, их сложнее отслеживать.
- Требуются:
    - централизованное логирование (ELK, Loki)
    - трассировка (Jaeger, Zipkin)
    - метрики и алерты (Prometheus, Grafana)
# 6. **Организационные вызовы**
- Команды должны быть **автономными и кросс-функциональными**.
- Требуется культура DevOps и CI/CD.
- Ошибки координации между командами → медленные релизы.
# 7. **Версионирование и совместимость API**
- Сервисы должны быть обратно совместимы, иначе сломается потребитель.
- Необходима стратегия:
    - семантическое версионирование API
    - feature toggles
    - деградация функционала
# 8. **Сложности тестирования**
- Интеграционное тестирование становится сложнее.
- Нужно использовать **контейнеры, мок-сервисы, тестовые брокеры сообщений**.
# 9. **Повышенные требования к DevOps и CI/CD**
- Каждый сервис нужно билдить, тестировать, деплоить отдельно.
- Автоматизация обязательна: Jenkins, GitHub Actions, GitLab CI, ArgoCD.
# 10. **Сложность при миграции данных**
- Базы данных тоже могут потребовать разбиения.
- Необходимо:
    - миграция схем
    - синхронизация данных между сервисами
    - поддержка согласованности
# ⭐ **Итог**
Переход на микросервисы даёт **масштабируемость, гибкость и автономию команд**, но:
- Увеличивает **сложность системы**
- Требует **новых практик DevOps и мониторинга**
- Меняет подход к **транзакциям, данным и API**
